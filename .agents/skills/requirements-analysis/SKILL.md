# Requirements Analysis

Use this skill to turn rough requirements into planning-ready analysis.

Follow `docs/agents/governance.md` as the durable workflow policy source.

## Procedure

1. Identify the actor, business goal, affected service desk module, and expected outcome.
2. Evaluate functional completeness.
3. Identify ambiguity, inconsistency, missing behavior, edge cases, and implicit assumptions.
4. Include UX, accessibility, security, performance, compliance, and data consistency risks when relevant.
5. Ask only clarification questions that can change planning or acceptance criteria.
6. Propose improved acceptance criteria.
7. Keep analysis within analyst scope; do not drift into implementation design unless a technical constraint must be named for clarity.

Use [the output format](./references/output-format.md) for the final artifact.
